Why you should use Version Control and Github if you've never heard of them

23 January 2015

What are the benefits of version control?

Version control is a way of saving and keeping track of changes to files in a project. At first that might not sound like much, after all most people make occasional copies of stuff they're working on anyway. But version control takes that idea a whole lot further. Version control makes it easier to revert back to some earlier state if something goes wrong.

The other great thing about version control is that it makes it HEAPS easier to share your work and collaborate with others.

Mostly version control gets used by programmers and developers but there are heaps of projects that can benefit from it. http://soundsoftware.ac.uk/why-version-control has a good flow diagram of whether or not you should consider using version control if you're not a programmer.

The reason programmers use version control is that software development is a surprisingly organic process. Projects grow in strange ways. Things branch off and flourish and need to be reincorporated. Or more likely they branch off and die and go nowhere. :) Humans are messy and need to be allowed to make mistakes to be creative. We need to iterate over problems to make progress. Version control helps you work on projects like this without going insane.

Git

Git is the flavour of version control system we'll look at but there are lots of others. See http://en.wikipedia.org/wiki/List_of_revision_control_software for a big old list.

How does git help you keep track of changes?

In version control you save your changes by doing a "commit." It's a snapshot of where things are up to. When you do a commit, you also record a message of what you think you've changed. There are guidelines for writing good commit messages because they're really important. If someone is looking back through your changes they can see your thinking and can compare your commit message with the actual changes they can see in your code. They're also very handy when you stop working on a project for a while and you need to work out what the hell you were doing when you left off.

You can view the log of changes as a text file but there are also handy tools (like in Github which we're about to get to) that let you visualise how things changed. This is really useful when things get more complicated and there are branches and merges and rebases and cherry picks and...yeah you get the idea.

Why use GitHub to store your code?

GitHub is a web service that you can use to store your projects that are controlled by git. When you store your project somewhere like GitHub it means others can contribute to the project in parallel. They can "clone" a copy of the code to their computer, do some work on it and then "push" it back. This is a bit of an over simplification, things get more complicated when there are bigger teams (They would probably create a pull request not just push it back) but that's the general idea.

Even if your project is personal, hosting it somewhere else means you get a version backed up somewhere should something bad happen to your local copy.

Where are all the sweet git commands I need to know? Why aren't you including them?

I think it's easy to scare people off with command line stuff. I resisted learning version control for this very reason. It seems scary and there's a lot of lingo which is intimidating.

If you're sold and want to have a play with git, I really got a lot from this site which has sort of gamified learning git commands. You can't break your computer or accidentally destroy an otherwise working project and it's really visual.

Once you get a little confidence with the basic lingo git's built in help is awesome. I keep forgetting git commands at the moment. I just type git with no arguments and you get a list of all of them. From there I can often work out how to build up my commands by typing "git help "
